一、项目全局概述

1.1 项目目标

基于 STM32H723VGT6 的 FEETECH STS3215 六轴串行舵机机械臂控制固件， 通过 USART10 串口总线驱动 6 个舵机实现关节运动控制， 并经 UART7 向 PC 输出调试日志。

2.5 Device/STS3215/STS3215_comm.c — 底层串口收发层

分层架构说明
Layer 1 (应用绑定): fputc() → printf → huart7 (调试) ftUart_Send() → HAL_UART_Transmit(huart10) (舵机TX) ftUart_Read() → HAL_UART_Receive(huart10) (舵机RX) ftBus_Delay() → HAL_Delay(1) ≈ 1ms Layer 2 (软件缓冲): wBuf[128] 全局发送缓冲区 (行 86) writeSCS() 写入wBuf，缓冲满时auto-flush (行 97) writeByteSCS() 单字节写入wBuf (行 116) wFlushSCS() 将wBuf一次性发送并清零 (行 139) rFlushSCS() 用timeout=0的HAL_UART_Receive循环清空RX (行 126) readSCS() 直接调用ftUart_Read (行 91)
  • 无 DMA：全部使用阻塞式 HAL_UART_Transmit/Receive，超时 100 ms
  • 软件发送缓冲：wBuf[128] 将零散写操作聚合为一次 UART 发送，减少总线空闲间隙
  • 接收无缓冲：readSCS() 直接阻塞接收，超时 100 ms 后返回 0
  • printf 重定向（行 12–16）：fputc → HAL_UART_Transmit(&huart7, ...) 100 ms 超时
⚠️ ftBus_Delay() 设置为 HAL_Delay(1) = 1 ms（行 71）， 远大于协议要求的 >10 µs，会显著降低总线利用率（浪费约 990 µs / 次）。

三、通信协议深度剖析

3.1 FEETECH STS3215 指令帧结构（根据代码还原）

发送帧（主机 → 舵机）

字节偏移0123456..NN+1
内容 0xFF0xFF IDLengthInstrMemAddr Data[0..]~CheckSum
  • 帧头：固定 0xFF 0xFF（writeBuf() 行 99–100）
  • ID：目标舵机 ID（1~253）；广播 = 0xFE
  • Length：后续字节数 = 参数字节数 + 2（含 Instr 和 CheckSum）
  • ~CheckSum：~(ID + Length + Instr + MemAddr + Data[0..N-1]) 低 8 位取反

应答帧（舵机 → 主机）

字节偏移012345..NN+1
内容 0xFF0xFF IDLengthStatus Data[0..]~CheckSum

写指令时 Length=2，无数据字节；~CheckSum = ~(ID + Length + Status + Data[0..N-1])（Ack() 行 401 / Read() 行 247）

3.3 SYNC_WRITE 相比逐一 WRITE 的优势（6 轴联动场景）

逐一 WRITE（6次）： ─[Write ID11]──[ACK11]──[Write ID12]──[ACK12]──...──[Write ID16]──[ACK16]─► 总时间 = 6 × T_frame，最后一轴比第一轴延迟约 5×T_frame SYNC_WRITE（1次广播）： ─[SyncWrite 0xFE, ALL 6 servos in ONE packet]──────────────────────────────► 总时间 = 1 × T_frame（无应答等待），所有舵机同一时刻收到指令
维度WRITE × 6SYNC_WRITE
总线占用时间O(N)O(1)
舵机启动时序一致性逐个延迟 ~T_frame几乎同时（<1 个位周期抖动）
应答开销需等 6 个 ACK无应答，无等待
关节抖动（jitter）可达几十 ms接近零
总线效率低（每帧 6 次完整往返）高（1 帧含所有数据）
同步启动❌ 不同步✅ 同步

代码体现（STS3215_control.c 行 96–129）：STS3215_SetPosEx_Sync() 将所有舵机的 ACC+Position+Time+Speed 打包为 offbuf[IDN*7]，一次 syncWrite() 广播， 所有舵机同帧触发运动。

四、硬件交互与调试设计

4.1 UART10 舵机 + UART7 调试分离的工程价值

对比维度合用单串口分离双串口

4.2 USART10 配置详解

usart.c 行 84–110 关键配置 
huart10.Init.BaudRate = 115200;          // ⚠️ 非1Mbps，舵机须配置为同波特率
huart10.Init.OverSampling = UART_OVERSAMPLING_16;  // 标准16x过采样
huart10.Init.OneBitSampling = UART_ONE_BIT_SAMPLE_DISABLE;  // 三样本多数决，更抗噪
huart10.Init.Mode = UART_MODE_TX_RX;     // 全双工，无半双工控制
// GPIO: PE2(RX) AF4_USART10 / PE3(TX) AF11_USART10
// 均配置为 GPIO_SPEED_FREQ_LOW（115200 bps 足够）
// FIFO 已禁用（HAL_UARTEx_DisableFifoMode，行 107）
// NVIC 已使能（行 197–198），但代码未使用 ISR 驱动舵机接收
注意： 当前波特率为 115200，与 STS3215 默认出厂波特率（1 Mbps）不同， 须舵机已预先配置为 115200（对应 SCS_115200=4）。 无半双工切换引脚，RX/TX 独立（PE2/PE3），与舵机连接时需关注总线拓扑。

4.3 其他外设

WS2812 RGB LED（Device/ws2812.c）

  • 使用 SPI6 主模式 TX-only（PA5=SCK, PA7=MOSI），SPI_PHASE_2EDGE
  • SPI 速率 ≈ 4.297 MHz（137.5 MHz / 32）
  • 每个 WS2812 bit 用 8 个 SPI bit 编码：0x60（0码）、0x78（1码），经 >>1 移位写入
  • 复位脉冲：发送 100 字节 0x00（行 61–64）
  • WS2812_Run() 随机色循环，含两次 rand_0_100() ms 随机延时 → 对主循环实时性有较大影响

按键与 LED（Device/Button_switch.c，Core/Src/gpio.c）

  • PA0：按键输入，上拉，低电平有效（GPIO_PIN_RESET）
  • PA2：LED 输出，上拉，初始低电平
  • Get_Button_State()：检测 PA0 低电平 → 消抖等待释放（busy-wait）→ 打印 → PA2 高（亮）1000 ms → PA2 低（灭）

4.4 启动等待 HAL_Delay(500) 的工程原因

原因说明
舵机电源稳定舵机独立供电，大电容充电需时间，未稳定时寄存器未就绪
舵机上电复位STS3215 内部 MCU 需完成自检、EEPROM 加载、通信初始化
避免总线冲突舵机上电瞬间总线可能输出不确定电平，干扰主控接收
提升 Find_STS3215() 可靠性500 ms 后所有舵机均已处于应答就绪状态，Ping 成功率高

随后在 STS3215_Init() 结束后还有 HAL_Delay(1000)（main.c 行 116）， 进一步确保舵机扫描完成。

影响：仅影响通过 FeedBack() 批量读后使用缓存的路径（ID==-1）；直接调用 readWord() 的路径（行 83）无此问题。
🐛 Bug 3：STS3215_SetPosEx_Sync() printf 格式串错误（行 130） 
printf("...: ID=%d, Position=%d, Speed=%d, ACC=%d\r\n", ID, Position, Speed, ACC);
// ID 是 uint8_t[]（数组指针），Position/Speed/ACC 是数组指针
// 将指针当整数打印 → 行为未定义
🐛 Bug 4：拼写错误 syncReadRxPacketToWrod()（Interface.h 行 35）

Wrod 应为 Word，出现在 STS3215_Interface.h 行 35 和 STS3215_Interface.c 行 500。

5.3 设计风险点

风险说明
全局 wBuf 非线程安全若引入 RTOS 则多任务并发会破坏缓冲区
无超时错误码体系u8Error 只有 4 个码，缺少分层错误传播
阻塞串口 100 ms 超时每帧最坏情况等待 100 ms，主循环可能卡住
Error_Handler() 挂死出错后 __disable_irq() + 死循环，无日志、无恢复
主循环含随机延时WS2812_Run() 延时 0~200 ms，舵机状态查询节率不稳定
HAL_Delay(1) 总线延时应为 >10 µs，实际 1 ms，每次操作多浪费约 990 µs
syncReadRxBuff 用 malloc嵌入式环境频繁动态内存分配有碎片风险
ID 扫描范围硬编码Find_STS3215() 固定扫 11~16，无法通过配置更改

5.4 可执行改进建议

短期（Bug 修复）

  • 修复 Find_STS3215() 中的死代码（else if (ID >= 0) 改为 else if (retry < 3)）
  • 修复 ReadPos/ReadSpeed/ReadLoad 缓存路径的字节序（改用 SCS2Host(Mem[L_off], Mem[H_off])）
  • 修复 STS3215_SetPosEx_Sync() 中的 printf 格式错误
  • 修正 syncReadRxPacketToWrod 拼写

中期（架构改善）

  • 分层错误码：定义 typedef enum STS_Result_t { STS_OK=0, STS_ERR_TIMEOUT, STS_ERR_CRC, STS_ERR_ID, ... }，所有函数统一返回
  • DMA + IDLE 接收：用 HAL_UARTEx_ReceiveToIdle_DMA() 替代阻塞接收，大幅提升吞吐
  • 状态机化总线管理：TX → 等待 turnaround → RX，替代当前的顺序阻塞调用
  • ftBus_Delay() 精确化：改用 DWT->CYCCNT 或 TIM 实现 10 µs 精确延时
  • 去掉主循环随机延时：WS2812_Run() 改为定时器中断驱动，与舵机控制解耦

长期（工程化提升）

  • 宏改枚举/结构体：寄存器地址用 typedef struct STS3215_Regs 代替散乱 #define
  • 参数范围校验：位置 0~4095，速度 0~3400 等在 API 入口检查
  • 单元测试框架：移植 CppUTest 或 Unity 对协议封包/解包做 host-side 测试
  • 动态 ID 配置：Find_STS3215() 扫描范围做成参数，结果存入全局 ID 列表
  • 日志分级：LOG_DEBUG/LOG_INFO/LOG_ERROR 宏，可通过编译开关关闭调试输出
  • 环形缓冲健壮性：test.c 中 uart_rx_idx 加临界区保护（__disable_irq/__enable_irq）
